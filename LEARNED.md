이건 "Harness(하네스)" 프레임워크 — Claude Code(에이전트)를 자율적으로 굴려서 프로젝트를 단계별로 구현하게 만드는 메타 프레임워크예요. 구조를 차근차근 설명해드릴게요.

  큰 그림: 이게 뭐 하는 물건인가

  핵심 아이디어는 이거예요:
  
  ▎ "사람이 한 번에 큰 작업을 시키면 LLM은 실수한다. 그러니 작업을 작고 자기완결적인 step들로 쪼개고, 각 step을 독립된 Claude 세션에서 실행하되, 매번 프로젝트 규칙(가드레일)을 강제 주입하고, 실패하면 자동
  ▎ 재시도하게 만들자."

  말(馬)에게 마구(馬具, harness)를 채워 정해진 방향으로만 달리게 하듯, Claude에게 가드레일을 채워 일관성 있게 코드를 생산하도록 통제하는 시스템이에요.

  폴더 구조와 각 파일의 역할

  harness/
  ├── CLAUDE.md              # 프로젝트 규칙 (템플릿 상태)
  ├── .gitignore             # step 실행 산출물(output.json) 제외
  ├── .claude/
  │   ├── settings.json      # 훅(hook) 정의 — 안전장치
  │   └── commands/
  │       ├── harness.md     # /harness 슬래시 커맨드 — 워크플로우 정의 (핵심!)
  │       └── review.md      # /review 슬래시 커맨드 — 변경사항 리뷰
  ├── docs/                  # 가드레일 문서 (전부 템플릿 상태, {중괄호}로 채워야 함)
  │   ├── PRD.md             # 제품 기획 (무엇을/누구에게/왜)
  │   ├── ARCHITECTURE.md    # 디렉토리 구조 + 데이터 흐름
  │   ├── ADR.md             # 기술 선택 결정 기록 (왜 이 기술을 골랐나)
  │   └── UI_GUIDE.md        # 디자인 규칙 (특히 "AI 슬롭 안티패턴" 금지 목록이 흥미로움)
  └── scripts/
      ├── execute.py         # 엔진 — step을 순차 실행/자가교정하는 오케스트레이터
      └── test_execute.py    # execute.py 안전망 테스트

  ▎ 참고: docs/의 문서들과 CLAUDE.md는 전부 빈 템플릿이에요 ({프로젝트명}, {기능 1} 처럼 중괄호 자리표시자). 즉 이 폴더는 "실제 프로젝트"가 아니라, 새 프로젝트에 복사해서 채워 쓰는 
  ▎ 보일러플레이트/스캐폴드예요.

  핵심 동작 흐름 (가장 중요한 부분)

  1단계 — /harness 커맨드 (설계 단계, 사람과 협업)

  commands/harness.md가 워크플로우 A~E를 정의해요:

  - A. 탐색: docs/를 읽고 기획·아키텍처 파악
  - B. 논의: 결정할 사항을 사용자와 상의
  - C. Step 설계: 작업을 여러 step으로 쪼갠 초안 작성
  - D. 파일 생성: 승인되면 phases/ 디렉토리에 실행 파일 생성
  - E. 실행: execute.py 실행

  여기서 step 설계 원칙 7가지가 이 프레임워크의 철학이에요:
  1. Scope 최소화 — 한 step = 한 레이어/모듈
  2. 자기완결성 — 각 step은 독립 세션에서 돌기 때문에 "아까 말했듯이" 같은 외부 참조 금지, 필요한 정보 전부 파일 안에
  3. 사전 준비 강제 — 읽어야 할 파일 경로를 명시해서 맥락부터 파악하게 함
  4. 시그니처 수준 지시 — 인터페이스만 주고 구현은 에이전트 재량, 단 핵심 규칙(멱등성·보안 등)은 못 박음
  5. AC는 실행 가능한 커맨드 — "동작해야 한다"(추상) ❌ → npm test(실행 가능) ✅
  6. 주의사항 구체적으로 — "조심해라" ❌ → "X 하지 마라, 이유: Y" ✅

  2단계 — phases/ 파일 구조 (실행 명세)

  /harness가 생성하는 것들:
  - phases/index.json — 전체 task 현황 (top-level)
  - phases/{task}/index.json — 각 step의 status 추적
  - phases/{task}/step0.md, step1.md, ... — 각 step의 실제 지시서 (읽을 파일 / 작업 / AC / 검증 / 금지사항)

  status는 pending → completed / error / blocked로 전이되고, 타임스탬프는 엔진이 자동 기록해요.

  3단계 — execute.py (엔진, 자율 실행)

  이게 진짜 심장부예요. python3 scripts/execute.py {task}를 돌리면:

  ┌──────────────────┬──────────────────────┬───────────────────────────────────────────────────────────────────────────────┐
  │       기능       │     코드상 위치      │                                     설명                                      │
  ├──────────────────┼──────────────────────┼───────────────────────────────────────────────────────────────────────────────┤
  │ 브랜치 자동 생성 │ _checkout_branch     │ feat-{task} 브랜치로 체크아웃                                                 │
  ├──────────────────┼──────────────────────┼───────────────────────────────────────────────────────────────────────────────┤
  │ 가드레일 주입    │ _load_guardrails     │ CLAUDE.md + docs/*.md를 매 step 프롬프트에 통째로 삽입                        │

  │ 컨텍스트 누적    │ _build_step_context  │ 완료된 step의 summary를 다음 step에 전달 (세션은 분리돼 있지만 요약은 이어짐) │

  - Stop 훅: 세션이 끝날 때마다 lint → build → test를 자동 실행

  ## 2
  ❯ 그럼 가드레일주입은 docs 안의 문서들을 보고 주입시키는거네?                                                                                                                     

⏺ 네, 맞아요. 정확히 짚었어요. 다만 한 가지만 더 정밀하게 보태면 — docs/ 안의 문서들 + 루트의 CLAUDE.md, 이 둘을 합쳐서 주입해요.

  _load_guardrails (177행)를 다시 보면:

  def _load_guardrails(self) -> str:
      sections = []
      claude_md = ROOT / "CLAUDE.md"          # ① 루트의 CLAUDE.md
      if claude_md.exists():
          sections.append(...)
      docs_dir = ROOT / "docs"
      if docs_dir.is_dir():
          for doc in sorted(docs_dir.glob("*.md")):   # ② docs/ 안의 모든 .md
              sections.append(...)
      return "...".join(sections)

  주입되는 것 = 두 곳

  ┌──────────────────┬──────────────────────────────────────────────────────────────────┐
  │       출처       │                               내용                               │
  ├──────────────────┼──────────────────────────────────────────────────────────────────┤
  │ CLAUDE.md (루트) │ 프로젝트 규칙 (기술 스택, CRITICAL 아키텍처 규칙, 개발 프로세스) │
  ├──────────────────┼──────────────────────────────────────────────────────────────────┤
  │ docs/*.md 전부   │ PRD, ARCHITECTURE, ADR, UI_GUIDE — 알파벳 순(sorted)으로         │
  └──────────────────┴──────────────────────────────────────────────────────────────────┘

  즉 "docs만"이 아니라 CLAUDE.md + docs의 모든 마크다운이에요.

  두 가지 중요한 디테일

  1) docs/ 안의 .md는 "전부, 자동으로" 들어가요

  특정 파일을 골라 넣는 게 아니라 glob("*.md")로 싹 다 가져와요. 그래서 만약 당신이 docs/에 새 문서(예: SECURITY.md)를 추가하면 — 코드를 한 줄도 안 고쳐도 다음 실행부터 자동으로 
  가드레일에 포함돼요. 반대로 docs/에 잡다한 메모를 넣으면 그것도 매 프롬프트에 다 끌려 들어가니 토큰을 잡아먹어요.

  2) "주입(가드레일)"과 "읽어라(step 지시)"는 별개예요 — 헷갈리기 쉬운 부분

  - 가드레일 주입: execute.py가 문서 내용을 통째로 프롬프트에 박아넣음 → Claude가 무조건 봄
  - step.md의 "읽어야 할 파일": 거기엔 문서 내용이 아니라 경로만 적혀 있음 (예: /docs/ARCHITECTURE.md를 읽어라) → Claude가 직접 Read 도구로 열어봄

  harness.md의 step 템플릿을 보면 "## 읽어야 할 파일"에 경로 리스트가 있죠? 그건 가드레일 주입과 다른 경로예요. 정리하면 Claude는 같은 문서를 두 방식으로 만나요:

  ┌──────────────────┬─────────────┬───────────────────────────────────┐
  │       방식       │    누가     │              무엇을               │
  ├──────────────────┼─────────────┼───────────────────────────────────┤
  │ 가드레일 주입    │ execute.py  │ 문서 본문을 프롬프트에 강제 삽입  │
  ├──────────────────┼─────────────┼───────────────────────────────────┤
  │ "읽어야 할 파일" │ Claude 본인 │ 경로를 보고 직접 파일을 열어 읽음 │
  └──────────────────┴─────────────┴───────────────────────────────────┘

  전자는 "안 볼 수 없게" 강제하는 거고, 후자는 "스스로 맥락을 파악하라"고 유도하는 거예요. 같은 docs 문서를 이중으로 챙기는 셈이죠.

  이해한 게 맞아요 — docs(+ CLAUDE.md)를 보고 주입하는 게 가드레일 주입이에요.
