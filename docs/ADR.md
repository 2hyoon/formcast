# Architecture Decision Records

## 철학
MVP 속도와 학습 가치를 함께 추구한다. 프로젝트의 핵심 가치(스키마 해석·검증 로직)는 직접 구현하고, 그 외 주변부는 최소 구현을 택한다. 외부 의존성을 최소화한다.

---

### ADR-001: 스키마 파싱·검증을 직접 구현 (라이브러리 미사용)
**결정**: `ajv`, `react-hook-form` 등을 쓰지 않고 resolver/validator를 직접 구현한다.
**이유**: 이 프로젝트의 존재 이유가 곧 스키마 해석·검증 로직이다. 라이브러리로 대체하면 핵심이 사라지고, 학습·포트폴리오 가치가 없어진다. 직접 구현해야 단위 테스트와 자가교정 루프가 의미를 갖는다.
**트레이드오프**: JSON Schema 전체 스펙을 지원하지 못한다. 그래서 범위를 평면 서브셋으로 좁힌다 (ADR-003).

### ADR-002: 테스트 러너로 Vitest 선택
**결정**: Jest 대신 Vitest를 쓰고, `test` 스크립트는 `vitest run --passWithNoTests`로 둔다.
**이유**: Next.js + TS 환경에서 ESM/TS 설정이 가볍고 순수 로직 단위 테스트에 빠르다. Stop 훅이 매 세션마다 `npm run test`를 실행하므로, 테스트가 아직 없는 step에서도 통과해야 한다 — `--passWithNoTests`가 이를 보장한다.
**트레이드오프**: Jest 생태계의 일부 도구와 호환되지 않을 수 있으나, 이 프로젝트 범위에서는 무관하다.

### ADR-003: 평면(1뎁스) 스키마 서브셋으로 범위 제한
**결정**: `string`/`number`/`boolean`/`enum` 타입과 기본 제약(required, min/max, length, pattern)만 지원한다. 중첩 객체·배열·`$ref`·조합 스키마는 제외한다.
**이유**: MVP를 1~2일 안에 완성 가능한 범위로 묶는다. 평면 구조만으로도 resolver·validator·렌더러의 레이어 분리를 충분히 보여줄 수 있다.
**트레이드오프**: 실무 스키마 일부를 다루지 못한다. 중첩 지원은 v2 확장 과제로 남긴다.
